# News for Students — 产品作品集

## 一、竞品分析

### 1.1 竞品矩阵

| 产品 | 类型 | 核心能力 | 不足 |
|------|------|----------|------|
| **Feedly** | RSS 聚合器 | 订阅源管理强大，信息覆盖面广 | 只有标题+摘要，无内容理解；不生成个人报告；纯英文阅读 |
| **Artifact** | AI 新闻推荐 | 个性化推荐算法好，摘要生成快 | 已关停（2024）；无法指定权威来源；无学习辅助功能 |
| **Perplexity** | AI 搜索引擎 | 搜索+总结结合好，支持追问 | 偏搜索工具，不产出结构化报告；无英语学习场景设计 |
| **Notion AI** | 文档助手 | 格式灵活，协作能力强 | 需要手动粘贴新闻；不是自动化工作流 |
| **一笔画 NEWS** | 微信小程序 | 短新闻+卡片式阅读，中文友好 | 内容由编辑挑选，非用户自主；无 AI 总结 |

### 1.2 竞品结论

现有产品有两种典型路径：要么是「新闻阅读器」（Feedly、一笔画），聚合信息但不做 AI 加工；要么是「通用 AI 问答」（Perplexity、ChatGPT），能做总结但缺少英语学习的结构化输出。

**空白地带**：没有一个产品把「权威来源 ➔ 真实搜索 ➔ AI 结构化报告 ➔ 英语学习场景」串成自动化工作流。这正是 News for Students 的切入点。

## 二、前期调研

### 2.1 用户需求分析

目标用户是**每天有英语新闻阅读任务的学生群体**。通过用户访谈归纳出四大痛点：

| 痛点 | 用户原话 | 对应功能 |
|------|----------|----------|
| 找新闻费时间 | "每天不知道去哪里找靠谱的英文新闻" | 预置权威来源（Reuters、BBC、The Guardian、TechCrunch），一键搜索 |
| 读完写不出总结 | "看完了不知道怎么写总结，总是抄原文" | AI 生成 2-3 句事实摘要，作为写作参考 |
| 个人反思难写 | "写 thoughts 很痛苦，想不出说啥" | AI 生成口语化反思，避免套话模板 |
| 第二天就忘了昨天看了什么 | "隔天忘了内容，感觉白看了" | 本地历史记录，随时回看 |

### 2.2 技术选型

| 层面 | 选型 | 理由 |
|------|------|------|
| **AI 编排** | Dify Workflow | 可视化工作流，支持条件分支，API 发布即用 |
| **新闻搜索** | Tavily Search API | 专为 AI Agent 设计，支持高级搜索+原始内容返回 |
| **大语言模型** | GPT-4o / Claude 3.5 Sonnet | JSON 输出稳定性好，多语言能力可靠 |
| **前端框架** | React 19 + Vite | 轻量、构建快、生态成熟 |
| **数据持久化** | localStorage | 个人小工具无需后端数据库，50 条上限够用 |

### 2.3 设计原则

1. **新闻真实性优先**：所有内容必须来自 Tavily 搜索结果，Prompt 明确禁止 LLM 编造事实
2. **反 AI 化写作**：personal_reflection 的 Prompt 专门设计"去 AI 味"，用第一人称、短句、口语化表达
3. **零运维负担**：前端纯静态部署，后端全靠 Dify Cloud 免费额度
4. **渐进式复杂度**：从单 LLM → 条件分支多风格，始终保留回退能力

## 三、制作过程

### 3.1 整体架构

```
┌──────────────────┐     ┌─────────────────────────────┐     ┌──────────────┐
│   React 前端      │ ──▶ │        Dify Workflow         │ ──▶ │  Tavily API  │
│ (Vite + React 19) │     │                              │     │  新闻搜索     │
│                   │ ◀── │ Start → HTTP → IF/ELSE → End │◀──  │              │
│ 选择来源/类别      │     │          ↓                   │     └──────────────┘
│ 生成按钮          │     │   ┌──────┼──────┐           │
│ 报告卡片           │     │   ▼      ▼      ▼           │
│ 历史记录           │     │ LLM-1  LLM-2  LLM-3        │
│ 双语切换           │     │ (严肃) (科技) (轻松)         │
│ 文件下载           │     └─────────────────────────────┘
└──────────────────┘
```

### 3.2 迭代历程

**V1 — 最小可行版本**
- 单个 LLM 工作流：Tavily 搜索 → LLM 生成报告 → 返回 JSON
- 前端完成来源/类别选择、生成按钮、报告卡片展示
- 发现 JSON 解析失败：LLM 会在 JSON 外包 ` ```json ` 代码块，前端补充了 `extractJson()` 自动剥离

**V1.5 — 下载与导出**
- .docx / PDF / Markdown 三种导出格式
- PDF 方案从 `window.print()`（整页截图）改为生成独立 HTML 页面，在新窗口触发打印

**V2 — 双语支持**
- Dify Prompt 改为输出双语 JSON（`_zh` 字段），一次生成中英文两份内容
- 前端报告卡片新增 `中`/`EN` 切换按钮，切换即时渲染，无需额外 API 调用
- 全局 UI 也支持中英文切换（`t` 翻译对象）

**V2.5 — 历史记录系统**
- localStorage 存储生成历史，最多 50 条
- 左侧侧边栏展示列表，点击回看，支持删除
- 响应式设计：桌面端侧边栏常驻，移动端汉堡菜单滑入

**V3 — 分风格工作流**
- Dify 引入 IF/ELSE 条件分支，根据新闻类别匹配不同 LLM Prompt：
  - Politics / Finance / Health → 严肃分析风
  - Technology → 创新关注风
  - Entertainment / Sports → 轻松活泼风
- 前端适配多分支输出：`outputs` 中可能有 `text`、`text_1`、`text_2`，自动扫描取第一个非空值

### 3.3 关键问题与解决

| 遇到的问题 | 解决方案 |
|------------|----------|
| LLM 日期输出"Mon DD, YYYY"占位符 | Prompt 写法从格式模板改为示例引导；前端兜底检测占位符关键词 |
| Dify 400：focus 字段不匹配 | 前端与 Dify Start 节点同步删掉无用变量 |
| 多个 LLM 分支导致 outputs 有 null 字段 | `parseDifyAnswer` 改为 `Object.values(outputs).find(v => v != null)` |
| Word/Markdown 下载文件日期缺失 | 统一走 `formatDate()` 兜底函数，HTML 内容加 `htmlEscape()` 防注入 |
| 生成完成无感知 | 添加绿色 Toast 弹窗动画，2.5 秒自动消失 |

### 3.4 项目总结

这是一个面向**真实个人场景**的 AI Agent 应用——不是为了做而做，而是每天自己要用。从单节点 Workflow 到分风格条件分支，从 JSON 解析崩溃到双语即时切换，踩过的坑都是 Dify + LLM 工程化的典型案例。

核心收获：
- **Prompt 工程是产品的一部分**：`personal_reflection` 的"去 AI 化"需要反复调试措辞，比写代码更花时间
- **不要过度设计**：一开始做了 Topic Focus、收藏按钮，实际用不到就果断删
- **条件分支是杠杆**：3 个 LLM 让三种类别的报告风格明显不同，工作流复杂度只多了一个 IF/ELSE 节点
